## Problem

The user saves content across X.com, LinkedIn, browser bookmarks, and other sources but cannot efficiently retrieve or reuse it. Content is scattered, untagged, and often lost. Primary usage is mobile, where the "Share" button is the dominant capture gesture.

---

## Solution

A Python agent with two capture modes converging into a single processing pipeline, with Notion as the knowledge base.

### Key Design Decision: LinkedIn
LinkedIn has no official API for saved posts. Instead of scraping, LinkedIn is handled via the **PUSH path**: the user shares the post URL to the Telegram bot directly. This is more reliable and requires no authentication.

## Processing Pipeline

```
URL received
    в”Ӯ
    в–ј
1. DEDUP CHECK (SQLite)
   Already processed? вҶ’ skip silently
    в”Ӯ
    в–ј
2. EXTRACTOR
   A. trafilatura.fetch_url()        вҶ’ articles, blogs, docs
   B. Playwright headless (fallback) вҶ’ JS-heavy / SPA sites
   C. Type detection:
      - tweet/thread вҶ’ Twitter API metadata
      - PDF          вҶ’ pdfplumber
      - Video        вҶ’ metadata only (title, description)
      - Paywall      вҶ’ flag, save metadata only
    в”Ӯ
    в–ј
3. CLASSIFIER (hybrid)
   Always (fast, local):
   - Language detection (langdetect)
   - Keywords (YAKE, top 5вҖ“8 terms)
   - Content type (heuristic rules)

   If text > 400 words (Claude Haiku):
   - 3вҖ“5 sentence summary
   - Enriched tags
   - Entities: people, tools, companies
    в”Ӯ
    в–ј
4. NOTION WRITER
   Creates structured page
   Marks URL as processed in SQLite
```

### Libraries

| Task | Library | Reason |
|---|---|---|
| Article extraction | `trafilatura` | Best accuracy for article text |
| JS-heavy sites | `playwright` | Full rendering |
| Keywords | `yake` | Multilingual, no training |
| Language detection | `langdetect` | Lightweight, covers ES/EN/PT |
| LLM classification | `anthropic` (Haiku) | ~$0.001/item |
| PDFs | `pdfplumber` | Structured text extraction |
| Scheduling | `apscheduler` | In-process cron |
| CLI | `click` | Ergonomic CLI framework |
| Config | `tomllib` + `python-dotenv` | Secrets separate from config |

## Notion Schema

**Database: "Knowledge Base"**

| Property | Type | Values / Notes |
|---|---|---|
| Title | title | Extracted or original page title |
| URL | url | Original URL |
| Source | select | twitter, linkedin, browser, telegram |
| Content Type | select | article, thread, video, paper, tool, doc |
| Language | select | es, en, pt |
| Tags | multi_select | Generated by classifier |
| Entities | multi_select | People, companies, technologies |
| Summary | rich_text | 3вҖ“5 sentence AI summary |
| Read Status | select | рҹ“Ҙ Inbox, рҹ‘Җ Reading, вң… Done, рҹ—ғпёҸ Archived |
| Relevance | select | вӯҗ Alta, рҹ”№ Media, в–«пёҸ Baja |

**Suggested views:**
- рҹ“Ҙ Inbox вҖ” unread, sorted by Saved At

## Configuration (`config/settings.toml`)

```toml
[telegram]
token = ""              # from env: TELEGRAM_BOT_TOKEN
allowed_user_id = 0     # your Telegram user ID (security)

[twitter]
bearer_token = ""       # from env: TWITTER_BEARER_TOKEN
oauth_client_id = ""

[notion]
api_key = ""            # from env: NOTION_API_KEY
database_id = ""

[classifier]
min_words_for_llm = 400   # threshold for Claude enrichment
claude_model = "claude-haiku-4-5"

[scheduler]
sync_cron = "0 8 * * *"   # daily at 8am
```

---

## Security

- Telegram bot only responds to `allowed_user_id` вҖ” ignores all other senders
- API keys loaded from environment variables, never from `settings.toml` directly
- `state.db` stored locally, never transmitted
